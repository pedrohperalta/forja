01 — Problema

PROBLEMA

Contexto Atual

O usuário treina 3x/semana seguindo um plano ABC de hipertrofia prescrito pelo instrutor. Hoje não existe nenhum registro digital dos treinos — pesos, séries e progresso ficam na memória ou em anotações desorganizadas.

Dor / Limitação Existente

Sem dados estruturados, é impossível implementar sobrecarga progressiva. O usuário repete os mesmos pesos semana após semana sem perceber. Apps existentes (Strong, Hevy, JEFIT) são complexos demais para quem segue um plano fixo simples.

Evidências

Validação direta com o usuário durante a discovery (9 sessões de Q&A documentadas). O problema foi confirmado como real e recorrente — a falta de registro é o principal obstáculo para progressão.

02 — Objetivo

OBJETIVO

Construir um app mobile pessoal em Expo (React Native) que permita registrar treinos de hipertrofia durante a sessão na academia. O app funciona 100% offline, com interface em português, e foi projetado para ser usado com uma mão entre séries.

03 — Solução Proposta

SOLUÇÃO PROPOSTA

App mobile construído com Expo SDK 55 (React Native 0.83) usando Expo Router para navegação file-based, Zustand v5 para gerenciamento de estado com persistência via AsyncStorage, e react-native-svg para o timer de descanso.

A arquitetura é propositalmente simples: um único repositório, sem backend, sem banco de dados externo. Toda a lógica vive no cliente. A camada de storage é abstraída para facilitar migração futura para sync com nuvem.

04 — Modelo de Dados

MODELO DE DADOS

O app trabalha com dados estáticos (planos de treino) e dados dinâmicos (registros do usuário). Os planos são hardcoded na v1. Os registros são persistidos via Zustand + AsyncStorage.

Plan (Plano de Treino) — Estático

type Plan = { id: 'A' | 'B' | 'C' name: string // "Peito / Ombros / Tríceps" focus: string // "7 exercícios · Hipertrofia" exercises: Exercise[] } type Exercise = { id: string // "supino-45-bilateral" name: string // "Supino 45° Bilateral" category: string // "EMPH" | "COMP" | "ISO" equipment: string // "Halter" | "Máquina" | "Barra" reps: string // "10-15" sets: number // 3 }

WorkoutSession — Dinâmico (histórico)

type WorkoutSession = { id: number // Date.now() timestamp planId: string // 'A' | 'B' | 'C' planName: string // "Peito / Ombros / Tríceps" focus: string date: string // "2026-03-20" durationMinutes: number exercises: CompletedExercise[] } type CompletedExercise = { name: string sets: number // quantas séries completou weight: number // último peso registrado (kg) }

05 — Navegação & Telas

NAVEGAÇÃO & TELAS

Navegação via Expo Router com file-based routing. A estrutura é um stack simples — sem tabs, sem drawer. Todas as telas de treino ativo ficam em um grupo (workout) que compartilha o mesmo layout sem header nativo.

Estrutura do Diretório app/

app/ ├── _layout.tsx // Root Stack (headerShown: false) ├── index.tsx // Home — lista de treinos + histórico ├── history.tsx // Histórico completo de treinos └── (workout)/ ├── _layout.tsx // Workout Stack (headerShown: false, gestureEnabled: false) ├── exercise.tsx // Tela de exercício — input de peso, séries ├── rest.tsx // Timer de descanso (60s) ├── checkpoint.tsx // Revisão de exercícios pulados └── complete.tsx // Resumo final do treino

Nota: Bloqueio de Gestos

O grupo (workout) desabilita gestureEnabled para impedir que o usuário saia do treino acidentalmente com swipe back. A única forma de sair é pelo botão de voltar na tela de exercício (com confirmação) ou completando o treino.

06 — Gerenciamento de Estado

GERENCIAMENTO DE ESTADO

Dois Zustand stores separados por responsabilidade. Ambos usam o persist middleware com createJSONStorage(() => AsyncStorage) para persistência automática. A separação permite que o workoutStore seja resetado ao final de cada treino sem afetar os dados de longo prazo do appStore.

workoutStore — Treino Ativo

Gerencia todo o estado transitório de um treino em andamento. Resetado ao concluir.

interface WorkoutState { // Estado activePlan: Plan | null queue: Exercise[] // fila de exercícios restantes skippedIds: Set<string> // IDs de exercícios pulados currentSet: number // série atual (1, 2 ou 3) currentSets: SetRecord[] // séries já feitas do exercício atual log: ExerciseLog[] // exercícios já concluídos startedAt: number | null // timestamp de início // Ações startWorkout(plan: Plan): void completeSet(weight: number): void skipExercise(): void removeExercise(): void // "não vou fazer hoje" returnToSkipped(id: string): void skipAllPending(): void reset(): void }

appStore — Dados Persistentes

Gerencia dados de longo prazo: últimos pesos por exercício, datas dos treinos, e histórico completo.

interface AppState { // Estado lastWeights: Record<string, number> // exerciseId → último peso (kg) lastDates: Record<string, string> // planId → última data ISO history: WorkoutSession[] // treinos concluídos // Ações saveWorkout(session: WorkoutSession): void updateLastWeights(weights: Record<string, number>): void deleteWorkout(id: number): void }

Nota: Hydration Assíncrona

Como AsyncStorage é assíncrono, o store não estará hidratado no primeiro render. Usamos o hook onRehydrateStorage do Zustand persist para exibir um splash/loading até a hidratação completar. O workoutStore também permite recuperar um treino interrompido (crash ou kill do app).

07 — Timer SVG

TIMER DE DESCANSO

O timer de descanso usa um arco circular SVG animado via react-native-svg + react-native-reanimated. A técnica usa strokeDasharray e strokeDashoffset para controlar o preenchimento do arco.

Implementação

  • Componente RestTimer recebe duration (60s padrão) e callback onComplete
  • Arco SVG: círculo com r=45, circumference = 2π×45 ≈ 282.74
  • strokeDashoffset animado de 0 → circumference via useSharedValue + withTiming
  • Últimos 10s: cor muda de #C2F000 (accent) para #FF453A (danger)
  • Ao chegar a 0: vibração via expo-haptics (foreground only na v1)
  • Botão "Pular Descanso" cancela o timer e navega para o próximo exercício

Comportamento em Background (v1)

Na v1, o timer usa Date.now() para calcular o tempo restante ao retornar do background. Se o timer expirou enquanto o app estava em background, o app navega diretamente para o próximo exercício. Notificações push locais estão planejadas para v2.

08 — Estratégia de Testes

ESTRATÉGIA DE TESTES E PUBLICAÇÃO

Testes Unitários

  • Zustand stores: testar todas as ações do workoutStore e appStore isoladamente (completeSet, skipExercise, removeExercise, saveWorkout, deleteWorkout)
  • Lógica de fila: testar cenários de skip + checkpoint (todos pulados → checkpoint, último exercício completo com pulados restantes)
  • Timer: testar cálculo de offset e comportamento de retorno de background

Testes End-to-End

  • Fluxo completo: Home → selecionar treino → completar 3 séries × 7 exercícios → resumo → home
  • Fluxo com skip: pular 2 exercícios → completar restantes → checkpoint → decidir sobre pendentes
  • Histórico: completar treino → verificar no histórico → apagar → confirmar exclusão

Publicação

  • Desenvolvimento: Expo Go para testes rápidos em device físico
  • Build: EAS Build (managed) para gerar binários iOS/Android
  • Distribuição: EAS Update para OTA updates, TestFlight (iOS) e internal testing track (Android) para beta

Plano de Rollback

Como o app é pessoal (um único usuário), rollback é simples: reverter o update via EAS ou reinstalar a versão anterior. Dados em AsyncStorage são preservados entre versões.

09 — Dívida Técnica

DÍVIDA TÉCNICA

Planos de treino hardcoded

Na v1, os 3 planos ABC são definidos em um arquivo data/plans.ts. Não há editor de planos. Quando o instrutor mudar o plano, será necessário alterar o código e fazer deploy. Migração planejada: editor de plano com persistência local (v2) e posteriormente sync com backend (v3).

Set<string> no Zustand persist

Zustand persist serializa estado como JSON, que não suporta Set nativamente. O skippedIds precisa de um custom serializer (Set → Array na serialização, Array → Set na hidratação) ou ser armazenado como array. Recomendação: usar array com helper isSkipped(id).

Timer sem notificação em background

Na v1, o timer não notifica o usuário em background. Se o usuário abrir outra app durante o descanso, não receberá alerta. Migração planejada: expo-notifications com scheduled notification local (v2).

Preparação para sync futuro

A decisão de usar Zustand persist + AsyncStorage na v1 foi consciente. Para facilitar a migração futura para sync com nuvem, a camada de dados está isolada em stores bem definidos. Na v2, um adapter pode substituir AsyncStorage por uma solução que faz sync (ex: Supabase, Firebase, ou API custom) sem alterar os componentes UI.
