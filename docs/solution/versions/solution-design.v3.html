01 — Problema

PROBLEMA

Contexto Atual

O usuário treina 3x/semana seguindo um plano ABC de hipertrofia prescrito pelo instrutor. Não existe nenhum registro digital — pesos, séries e progresso ficam na memória ou em anotações desorganizadas.

Dor / Limitação Existente

Sem dados estruturados, é impossível implementar sobrecarga progressiva. Apps existentes (Strong, Hevy, JEFIT) são complexos demais para quem segue um plano fixo simples.

Evidências

Validação direta com o usuário durante a discovery (9 sessões de Q&A documentadas). A falta de registro é o principal obstáculo para progressão.

03 — Solução Proposta

SOLUÇÃO PROPOSTA

App construído com Expo SDK 55 usando New Architecture exclusivamente (legacy bridge removido). Navegação file-based via Expo Router em src/app/, estado com Zustand v5 persistido via MMKV (20-30x mais rápido que AsyncStorage, leitura síncrona), estilização com NativeWind (Tailwind CSS), e o React Compiler v1.0 elimina necessidade de memoização manual.

New Architecture (Obrigatória no SDK 55)

  • JSI substitui a bridge JSON-serialized (comunicação direta JS ↔ C++)
  • Fabric renderiza UI com layout síncrono
  • TurboModules carregam funcionalidades nativas lazy (menos startup time)
  • ~43% melhoria cold start, ~39% rendering speed, ~20-30% redução memória

React Compiler v1.0

Memoização automática em build time via Babel plugin. Sem necessidade de useMemo/useCallback/React.memo manual — o compiler insere cache boundaries por reactive scope. Componentes devem seguir Rules of React (sem mutação de props, sem leitura de estado mutável externo durante render).

04 — Modelo de Dados

MODELO DE DADOS

Dados estáticos (planos de treino, hardcoded na v1) e dinâmicos (registros do usuário). Persistência via Zustand persist + MMKV StateStorage adapter. Entidades já incluem sync metadata (syncStatus, version, timestamps) preparando para cloud sync na v2.

Branded Types (zero runtime cost)

type Brand<K, T> = K & { __brand: T } type ExerciseId = Brand<string, 'ExerciseId'> type WorkoutId = Brand<string, 'WorkoutId'> type PlanId = Brand<string, 'PlanId'>

Plan (Plano de Treino) — Estático

type Plan = { id: PlanId // 'A' | 'B' | 'C' name: string // "Peito / Ombros / Tríceps" focus: string // "7 exercícios · Hipertrofia" exercises: Exercise[] } type Exercise = { id: ExerciseId // "supino-45-bilateral" name: string // "Supino 45° Bilateral" category: string // "EMPH" | "COMP" | "ISO" equipment: string // "Halter" | "Máquina" | "Barra" reps: string // "10-15" sets: number // 3 } const PLANS = { A: { ... }, B: { ... }, C: { ... }, } satisfies Record<string, Plan> // preserves literal types

WorkoutSession — Dinâmico (com sync metadata)

type WorkoutSession = { id: WorkoutId // Date.now() timestamp planId: PlanId planName: string focus: string date: string // "2026-03-20" durationMinutes: number exercises: CompletedExercise[] // sync metadata (prepared for v2) syncStatus: 'local' | 'synced' | 'pending' version: number createdAt: string // ISO timestamp updatedAt: string // ISO timestamp } type CompletedExercise = { name: string sets: number weight: number // último peso registrado (kg) }

Zod Schemas (validação runtime na boundary de storage)

import { z } from 'zod' const WorkoutSessionSchema = z.object({ id: z.string(), planId: z.enum(['A', 'B', 'C']), date: z.string(), durationMinutes: z.number(), exercises: z.array(z.object({ name: z.string(), sets: z.number(), weight: z.number(), })), syncStatus: z.enum(['local', 'synced', 'pending']), version: z.number(), }) // Always use safeParse, never parse const result = WorkoutSessionSchema.safeParse(data)

MMKV StateStorage Adapter

05 — Navegação & Telas

NAVEGAÇÃO & TELAS

Expo Router com file-based routing em src/app/. Stack simples, sem tabs. Telas de treino ativo agrupadas em (workout) com gestos desabilitados. Path aliases @/ → ./src/ configurados em tsconfig.json e babel.config.js. Typed routes habilitadas via experiments.typedRoutes.

06 — Gerenciamento de Estado

GERENCIAMENTO DE ESTADO

Dois Zustand stores (slice pattern) com persist middleware + MMKV adapter. Leitura síncrona (sem async hydration). Middleware order: persist(devtools(...)). Selectors atômicos obrigatórios: useStore(s => s.bears). Para múltiplos valores: useShallow. Schema versioning (version + migrate) configurado desde o dia um.

workoutStore — Treino Ativo

interface WorkoutState { // Estado activePlan: Plan | null queue: Exercise[] // fila de exercícios restantes skippedIds: string[] // IDs pulados (array, not Set — JSON safe) currentSet: number // série atual (1, 2 ou 3) currentSets: SetRecord[] // séries já feitas do exercício atual log: ExerciseLog[] // exercícios já concluídos startedAt: number | null // Ações startWorkout(plan: Plan): void completeSet(weight: number): void skipExercise(): void removeExercise(): void // "não vou fazer hoje" returnToSkipped(id: ExerciseId): void skipAllPending(): void isSkipped(id: ExerciseId): boolean // helper reset(): void } // Persist config { name: 'workout-store', storage: createJSONStorage(() => mmkvStorage), version: 1, partialize: (state) => ({ // only persist essential state, not ephemeral UI activePlan: state.activePlan, queue: state.queue, skippedIds: state.skippedIds, currentSet: state.currentSet, currentSets: state.currentSets, log: state.log, startedAt: state.startedAt, }), }

appStore — Dados Persistentes

interface AppState { lastWeights: Record<string, number> // exerciseId → último peso (kg) lastDates: Record<string, string> // planId → última data ISO history: WorkoutSession[] saveWorkout(session: WorkoutSession): void updateLastWeights(weights: Record<string, number>): void deleteWorkout(id: WorkoutId): void } // Persist config { name: 'app-store', storage: createJSONStorage(() => mmkvStorage), version: 1, migrate: (persisted, version) => { /* schema evolution */ }, }

Padrão de Selectors (Zustand v5)

// ✅ Atomic selector — single value const queue = useWorkoutStore(s => s.queue) // ✅ Multiple values — useShallow const { queue, currentSet } = useWorkoutStore( useShallow(s => ({ queue: s.queue, currentSet: s.currentSet })) ) // ❌ NEVER — new object without useShallow causes infinite re-renders const { queue, currentSet } = useWorkoutStore( s => ({ queue: s.queue, currentSet: s.currentSet }) )
07 — Timer SVG

TIMER DE DESCANSO

Arco circular SVG animado via react-native-svg + Reanimated 4. Usa useAnimatedProps (não useAnimatedStyle) para animar props SVG na UI thread. Animated.createAnimatedComponent(Circle) para componente animável.

Implementação

  • Componente RestTimer recebe duration (60s) e callback onComplete
  • Arco SVG: r=45, circumference = 2π×45 ≈ 282.74
  • strokeDashoffset animado de circumference → 0 via withTiming (linear)
  • transform="rotate(-90)" para início às 12h (não às 3h)
  • strokeLinecap="round" para pontas arredondadas
  • Últimos 10s: cor accent → danger
  • Ao chegar a 0: expo-haptics → notificationAsync(Success)
  • Countdown warnings (3-2-1): impactAsync(Light)

Background (v1)

Na v1, o timer usa Date.now() para calcular restante ao retornar de background. Se expirou, navega para próximo exercício. Notificações push locais planejadas para v2.

Acessibilidade do Timer

  • accessibilityRole="timer" com accessibilityLabel dinâmico
  • Progress ring: accessibilityRole="progressbar" com accessibilityValue={{ min: 0, max: 60, now: remaining }}

09 — Estratégia de Testes

11 — Dívida Técnica

DÍVIDA TÉCNICA

Planos de treino hardcoded

Na v1, os 3 planos ABC são definidos em src/constants/plans.ts com satisfies. Quando o instrutor mudar o plano, será necessário deploy. Migração: editor de plano com persistência local (v2) e sync backend (v3).

Timer sem notificação em background

Na v1, timer não notifica em background. Migração: expo-notifications com scheduled notification local (v2).

Sync metadata preparado desde v1

Todas as entidades dinâmicas já incluem syncStatus, version, createdAt, updatedAt. Na v2, um adapter pode substituir MMKV por uma solução que faz sync (Supabase, Firebase, ou API custom) sem alterar componentes UI. Zustand version + migrate configurados desde o início para schema evolution.

Fase 1 — Setup do Projeto

  • Instalar dependências

    ESLint flat config, Prettier, Lefthook, commitlint, path aliases (@/), typed routes, React Compiler Babel plugin
  • Configurar NativeWind com design tokens Forja
    tailwind.config.ts com cores (#080808 bg, #C2F000 accent, etc.), fontes (Bebas Neue, Syne), dark mode
  • Criar src/constants/plans.ts com os 3 planos ABC
    21 exercícios (7 por plano) com branded types, satisfies Record<string, Plan>
  • Criar MMKV StateStorage adapter
    Adapter para Zustand createJSONStorage, in-memory mock para testes

Fase 2 — Stores Zustand (TDD)

  • Escrever testes do appStore
    Cenários: saveWorkout, deleteWorkout, updateLastWeights, persistência via MMKV mock
  • Implementar appStore

  • Escrever testes do workoutStore

  • Implementar workoutStore
