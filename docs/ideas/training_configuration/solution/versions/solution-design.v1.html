Problema

Treinos fixos impedem o uso real do app

O Forja possui 3 planos de treino (A, B, C) com 21 exercícios definidos diretamente no código-fonte (src/constants/plans.ts). Para alterar qualquer parâmetro — trocar um exercício, ajustar séries, mudar o split — é necessário modificar código e reconstruir o app. Isso torna o Forja inutilizável como ferramenta de treino real, onde mudanças acontecem a cada mesociclo ou avaliação com personal.

Objetivo

Permitir configuração completa de treinos pelo usuário

  • Criar, editar e remover qualquer número de dias de treino
  • Adicionar exercícios com nome livre, configurando séries, repetições, descanso, categoria (seletor) e equipamento
  • Reordenar exercícios via drag-and-drop
  • Manter histórico imutável — edições não alteram treinos já realizados
  • Migrar automaticamente os 3 planos fixos existentes na primeira abertura
  • Bloquear configuração durante treino ativo
Solução Proposta

Arquitetura da Solução

Visão Geral

A solução adiciona um novo store dedicado (planStore) para gerenciar planos de treino mutáveis, 3 novas telas de configuração, e ajustes pontuais na home e no fluxo de treino existente para consumir planos dinâmicos em vez de constantes.

Modelo de Dados

Exercise ALTERADO

type Exercise = {
  id: ExerciseId          // UUID gerado na criação
  name: string            // nome livre digitado pelo usuário
  category: string        // seletor: Peito, Costas, Ombros, etc.
  equipment: string       // texto livre: Halter, Máquina, etc.
  reps: string            // range: "10-12" ou número: "15"
  sets: number            // número de séries
  restSeconds: number     // ← NOVO — tempo de descanso (padrão: 60)
}

Campo restSeconds adicionado. Valor padrão 60. Backwards-compatible — planos migrados recebem 60 automaticamente.

Plan ALTERADO

type Plan = {
  id: PlanId              // UUID (branded type)
  label: string           // indicador visual: "A", "B", "C", "D"...
  name: string            // nome dado pelo usuário: "Treino A"
  focus: string           // foco muscular: "Peito / Ombros / Tríceps"
  exercises: Exercise[]   // lista ordenada
}

Campo label adicionado — letra sequencial para exibição na UI (A, B, C...). O id passa a ser UUID em vez de letra simples.

PlanState (novo store) NOVO

interface PlanState {
  plans: Plan[]
  nextLabel: string       // próxima letra disponível ("D", "E"...)

  // Ações — planos
  addPlan: (name: string, focus: string) => PlanId
  updatePlan: (id: PlanId, changes: Partial<Pick<Plan, 'name' | 'focus'>>) => void
  removePlan: (id: PlanId) => void

  // Ações — exercícios
  addExercise: (planId: PlanId, exercise: Omit<Exercise, 'id'>) => void
  updateExercise: (planId: PlanId, exerciseId: ExerciseId, changes: Partial<Exercise>) => void
  removeExercise: (planId: PlanId, exerciseId: ExerciseId) => void
  reorderExercises: (planId: PlanId, orderedIds: ExerciseId[]) => void
}

Store dedicado com persistência MMKV independente. Chave: 'plan-store'. Versão: 1.

Migração dos Planos Fixos

Na primeira inicialização do planStore, se a lista de planos estiver vazia:

  1. Importar os 3 planos de constants/plans.ts
  2. Gerar UUID para cada plano, preservando label como "A", "B", "C"
  3. Adicionar restSeconds: 60 em cada exercício
  4. Definir nextLabel = "D"
  5. Persistir no MMKV

A constante PLANS permanece no código como fonte da migração. Pode ser removida após 2 releases.

Estratégia de IDs

EntidadeFormatoGeração
PlanId UUID v4 (branded) crypto.randomUUID() ou fallback Date.now().toString(36)
ExerciseId UUID v4 (branded) Mesmo mecanismo. IDs de exercícios migrados permanecem os originais (kebab-case).
Plan.label Letra sequencial ("A", "B"...) Incrementa a partir de nextLabel no store. Nunca reutiliza letras de planos deletados.
WorkoutId ${planId}-${startedAt} Sem mudança — usa o novo UUID do plano.

Impacto da Mudança de PlanId no Histórico

O appStore usa PlanId em dois locais:

  • lastDates: Record<PlanId, string> — rastreia última data de cada plano
  • history[].planId — referência no WorkoutSession

Na migração, ao gerar UUIDs para os planos, precisamos também atualizar:

  1. lastDates: remapear chaves de "A"/"B"/"C" para os novos UUIDs
  2. history: atualizar planId em cada WorkoutSession existente para o novo UUID correspondente

Essa migração ocorre uma vez, de forma atômica, no mesmo momento em que os planos são migrados para o planStore.

Bloqueio Durante Treino Ativo

Quando workoutStore.status === 'active':

  • O botão "Meus Treinos" na home fica desabilitado (opacity + disabled)
  • Se o usuário navegar diretamente para /plans, é redirecionado para home
  • Mesma estratégia de guard usada no layout (workout)/_layout.tsx

Timer de Descanso Dinâmico

Atualmente REST_DURATION_SECONDS = 60 é uma constante no módulo rest.tsx.

Mudança:

  • O workoutStore já tem acesso ao exercício atual via getCurrentExercise(queue, skippedIds)
  • Adicionar um selector currentRestSeconds que retorna currentExercise.restSeconds ?? 60
  • A tela rest.tsx lê esse valor do store em vez de usar constante
  • O hook useRestTimer já aceita durationSeconds como parâmetro — nenhuma mudança necessária no hook

Mudanças na Home

  • Fonte de dados: trocar Object.values(PLANS) por planStore.plans
  • Botão de configuração: adicionar botão "Meus Treinos" (pill, secondary) acima da lista de cards
  • Estado vazio: quando plans.length === 0, exibir mensagem de boas-vindas com CTA para /plans
  • getNextPlanId: atualizar utilitário para trabalhar com array de planos em vez de chaves fixas A/B/C

Seletor de Categorias

Lista fixa de 12 categorias musculares como constante em src/constants/categories.ts:

export const MUSCLE_CATEGORIES = [
  'Peito', 'Costas', 'Ombros', 'Bíceps', 'Tríceps',
  'Quadríceps', 'Posterior', 'Glúteos', 'Panturrilha',
  'Abdômen', 'Antebraço', 'Trapézio',
] as const

Renderizado como chips selecionáveis (pill shape) no formulário de exercício. Seleção única.

Drag-and-Drop para Reordenação

Usar react-native-draggable-flatlist na tela de detalhe do treino (/plans/[id]).

  • Ativação por press longo no handle de drag (ícone ⠿)
  • Compatível com Reanimated 4 e React Native New Architecture
  • Ao soltar, chamar planStore.reorderExercises(planId, newOrder)
  • Haptic feedback (light) ao iniciar drag via useHaptics

Dívida Técnica

O que introduzimos ou resolvemos

Resolvida

  • Planos hardcoded — a constante PLANS era a única fonte de dados. Agora planos vivem em store persistido, eliminando a necessidade de rebuild para qualquer mudança.
  • Rest timer fixo — os 60s hardcoded passam a ser configuráveis por exercício.

Introduzida

  • Constante PLANS como fonte de migração — mantida temporariamente para migração de usuários existentes. Remover após 2 releases.
  • Remapeamento de PlanIds no histórico — migração de IDs simples para UUID afeta appStore. Complexidade pontual necessária para manter integridade.
  • Dependência: react-native-draggable-flatlist — nova dependência externa para reordenação. Bem mantida e compatível com o stack atual.

Impacto no Produto

O que muda para o usuário

Ganhos

  • App se torna realmente utilizável — treinos refletem a realidade do usuário
  • Flexibilidade total: de 2-day split a 6-day PPL
  • Descanso personalizado por exercício melhora a experiência durante o treino
  • Migração transparente — zero fricção para usuários existentes

Riscos para o Usuário

  • Migração falha silenciosamente → usuário perde seus planos
  • Edição acidental durante treino → inconsistência entre plano e sessão

Ambos mitigados: migração testada extensivamente + configuração bloqueada durante treino ativo.
