Problema

Interações com o celular durante o treino criam fricção desnecessária

O Forja exige que o usuário interaja com o celular para cada ação do treino — concluir série, ver timer de descanso, pular exercício. Em um treino típico com 6 exercícios × 3 séries + 18 intervalos de descanso, são 36+ interações: tirar do bolso, desbloquear, tocar, guardar. Cada interação leva ~5 segundos e quebra o ritmo do treino.

Timer invisível

O timer de descanso roda no celular dentro do bolso. O usuário precisa sacar o aparelho repetidamente para ver quanto falta — ou simplesmente ignora e perde a cadência.

Mãos ocupadas

Entre séries, as mãos estão suadas e segurando halteres ou barras. Pegar o celular é desajeitado e arriscado (pode cair, escorregar).

Objetivo

Reduzir interações com o celular durante o treino a praticamente zero

Estender o Forja ao Samsung Galaxy Watch como display remoto + controle. O relógio espelha o treino ativo e permite as 3 ações mais frequentes — concluir série, ver timer e pular exercício. Toda a lógica de negócio permanece no celular; o relógio é intencionalmente "burro".

Solução Proposta

Arquitetura de alto nível

O sistema é composto por dois apps — o Forja existente (React Native / Expo) e um app nativo Wear OS (Kotlin / Jetpack Compose) — conectados via Wear OS Data Layer API. A comunicação é bidirecional: o celular publica estado, o relógio envia comandos.

Decisões técnicas

App nativo Kotlin (não React Native)

React Native não tem suporte oficial para Wear OS. O runtime RN depende de APIs Android indisponíveis no ambiente do relógio. Jetpack Compose for Wear OS é o padrão recomendado pelo Google, com componentes otimizados para tela circular e input simplificado.

Wear OS Data Layer API

API oficial do Google para comunicação celular ↔ relógio. Opera via Bluetooth (ou Wi-Fi quando disponível). Dois mecanismos:

  • DataClient: Sincroniza estado via DataItems. Persiste automaticamente — se o relógio reconectar, recebe o último estado. Ideal para estado do treino.
  • MessageClient: Fire-and-forget para comandos pontuais. Latência ~50-150ms via Bluetooth. Ideal para ações do usuário (concluir série, pular).

Timer local no relógio

O celular envia restSeconds quando o timer inicia. O relógio roda seu próprio CountDownTimer localmente — sem depender de atualizações a cada segundo via Bluetooth. Isso garante countdown suave mesmo com oscilações na conexão.

Native Module como bridge

Um módulo Kotlin exposto ao React Native via TurboModule (New Architecture). O módulo escuta mudanças no workoutStore via subscriber JS e publica no DataClient. Recebe comandos do MessageClient e invoca callbacks JS para dispatch de actions no store.

Protocolo de Dados

Formato das mensagens

Estado do treino (celular → relógio) DataClient

Publicado em /workout-state a cada mudança relevante no workoutStore:

// Path: /workout-state
{
  "status": "active" | "idle" | "completed",
  "screen": "exercise" | "rest" | "checkpoint" | "complete",

  // Exercise context (when screen = "exercise")
  "exerciseName": "Supino Reto",
  "exerciseIndex": 2,        // 0-based
  "totalExercises": 6,
  "currentSet": 2,           // 1-based
  "totalSets": 4,
  "reps": "10-12",
  "lastWeight": 32.0,        // kg, from appStore.lastWeights

  // Rest context (when screen = "rest")
  "restSeconds": 60,

  // Checkpoint context (when screen = "checkpoint")
  "skippedNames": ["Leg Press", "Extensora"],

  // Progress
  "setsCompleted": [true, true, false, false],

  "timestamp": 1712150400000
}

O DataClient persiste o último DataItem. Se o relógio reconectar após uma queda de conexão, recebe automaticamente o estado mais recente sem ação do celular.

Comandos (relógio → celular) MessageClient

Enviados como mensagens fire-and-forget no path /workout-command:

// Path: /workout-command
{ "action": "completeSet" }
{ "action": "skip" }
{ "action": "returnToSkipped", "exerciseId": "abc-123" }
{ "action": "returnAllSkipped" }

O bridge no celular traduz cada comando para a action correspondente do workoutStore. O completeSet usa o último peso registrado (appStore.lastWeights[exerciseName]) — sem input de peso no relógio.

Eventos de lifecycle (celular → relógio) MessageClient

// Path: /workout-lifecycle
{ "event": "workoutStarted", "planName": "Treino A" }  // → auto-launch watch app
{ "event": "workoutCompleted" }                         // → show summary, close
{ "event": "workoutReset" }                             // → return to idle

    private fun startLocalTimer(seconds: Int) {
        countDownTimer?.cancel()
        countDownTimer = object : CountDownTimer(seconds * 1000L, 1000) {
            override fun onTick(millisUntilFinished: Long) {
                timerSecondsLeft.value = (millisUntilFinished / 1000).toInt()
                timerProgress.value = 1f - (millisUntilFinished.toFloat() / (seconds * 1000))
            }
            override fun onFinish() {
                timerSecondsLeft.value = 0
                timerProgress.value = 1f
                vibratePattern()  // 2 curtos + 1 longo
            }
        }.start()
    }
}

    override fun onMessageReceived(messageEvent: MessageEvent) {
        if (messageEvent.path == "/workout-command") {

JavaScript side Novo

// src/services/wearBridge.ts
import { NativeModules, NativeEventEmitter } from 'react-native'

const { WearBridge } = NativeModules
const emitter = new NativeEventEmitter(WearBridge)

/** Subscribe to workoutStore and sync state to watch */
export function startWatchSync(store: typeof useWorkoutStore) {
  return store.subscribe((state) => {
    const watchState = mapToWatchState(state)
    WearBridge.syncWorkoutState(JSON.stringify(watchState))
  })
}

/** Listen for commands from watch and dispatch to store */
export function listenForWatchCommands(
  workoutStore: typeof useWorkoutStore,
  appStore: typeof useAppStore,
) {
  return emitter.addListener('onWatchCommand', (commandJson: string) => {
    const { action, exerciseId } = JSON.parse(commandJson)
    const state = workoutStore.getState()

    switch (action) {
      case 'completeSet': {
        const exerciseName = state.queue[0]?.name
        const lastWeight = appStore.getState().lastWeights[exerciseName] ?? 0
        workoutStore.getState().completeSet(lastWeight)
        break
      }
      case 'skip':
        workoutStore.getState().skipExercise()
        break
      case 'returnToSkipped':
        workoutStore.getState().returnToSkipped(exerciseId as ExerciseId)
        break
      case 'returnAllSkipped':
        state.skippedIds.forEach(id =>
          workoutStore.getState().returnToSkipped(id)
        )
        break
    }
  })
}

Resiliência

Comportamento offline e reconexão

DataClient persiste automaticamente

O DataClient do Wear OS armazena o último DataItem no dispositivo. Se a conexão cai e volta, o relógio recebe o estado mais recente automaticamente via onDataChanged — sem necessidade de lógica de reconexão manual.

Timer continua rodando

O CountDownTimer do relógio é local. Uma vez iniciado com restSeconds, continua rodando independente da conexão Bluetooth. A vibração ao final funciona mesmo offline.

Comandos perdidos

Se o relógio enviar um comando (completeSet) durante uma queda de conexão, o MessageClient não garante entrega. O relógio deve detectar que o estado não mudou após ~2s e exibir um toast "Sem conexão — tente novamente". O celular continua funcionando normalmente como fallback.

Deploy

Instalação no relógio

Via ADB Wi-Fi (recomendado para uso pessoal)

O app do relógio é buildado e instalado diretamente, sem Play Store:

# 1. Build do APK (no diretório watch/)
cd watch && ./gradlew assembleRelease

# 2. Conectar ao relógio (habilitar ADB Wi-Fi nas configurações)
adb connect 192.168.1.XX:5555

# 3. Instalar
adb install app/build/outputs/apk/release/app-release.apk

# 4. Atualizar (substitui sem desinstalar)
adb install -r app/build/outputs/apk/release/app-release.apk

Script de conveniência

# scripts/deploy-watch.sh
#!/bin/bash
set -e

WATCH_IP="${1:-192.168.1.42}"
echo "Building watch APK..."
cd watch && ./gradlew assembleRelease

echo "Connecting to watch at $WATCH_IP..."
adb connect "$WATCH_IP:5555"

echo "Installing..."
adb install -r app/build/outputs/apk/release/app-release.apk

echo "✅ Done! Forja Watch installed."

Dívida Técnica

Trade-offs aceitos

Sem input de peso no relógio

Digitar peso em tela circular de 1.4" é impraticável. O completeSet do relógio sempre usa o último peso registrado via appStore.lastWeights. Para ajustar o peso, o usuário precisa usar o celular. Aceitável pois a maioria das séries usa o mesmo peso.

Sem modo standalone

O relógio não funciona sem o celular. Toda lógica de negócio está no workoutStore. Implementar standalone exigiria duplicar stores e MMKV no Kotlin — complexidade desproporcional para app pessoal.

Projeto Gradle separado (não multi-module)

O app do relógio vive em watch/ como projeto Gradle independente, não como módulo do projeto principal do Expo. Isso simplifica o build (sem interferir no eas build) mas significa dois builds separados. Em v2, poderia integrar como multi-module se for publicar na Play Store.

Comandos não são enfileirados

Se a conexão cai no momento exato de um completeSet, o comando é perdido. O relógio detecta via timeout e pede retry. Para app pessoal na academia (Bluetooth range ~10m), a probabilidade é muito baixa.
