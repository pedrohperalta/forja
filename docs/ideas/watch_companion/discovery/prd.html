Visão Geral

Forja no Pulso

Durante o treino, cada interação com o celular é uma fricção. Para ver o timer de descanso, marcar uma série como concluída ou conferir o próximo exercício, o usuário precisa tirar o celular do bolso, desbloquear, e navegar até a tela certa. Em um treino típico com 18 séries e 18 períodos de descanso, são 36+ interações com o celular — cada uma interrompendo o ritmo do treino.

Com o Watch Companion, o Forja se estende ao Samsung Galaxy Watch. O relógio espelha o estado do treino ativo e permite as ações mais frequentes — ver o countdown do timer de descanso, concluir uma série e pular exercícios — tudo com um olhar rápido no pulso. O celular continua sendo o dispositivo principal; o relógio é o controle remoto.

Problema

Por que isso importa

Celular no bolso durante exercícios

Com as mãos ocupadas por barras, halteres ou cabos, tirar o celular do bolso entre séries é desajeitado. As mãos suam, o celular escorrega, e o processo de desbloquear e navegar até a tela certa leva vários segundos a cada interação.

Timer de descanso invisível

O timer de descanso é uma das funcionalidades mais usadas do Forja, mas só é visível na tela do celular. O usuário precisa ficar checando o celular repetidamente para saber quanto tempo falta — ou simplesmente ignora o timer e descansa "no feeling", perdendo o controle da cadência do treino.

Fluxo do treino é interrompido

Cada vez que o usuário pausa para interagir com o celular, o foco no exercício é quebrado. Em um treino de 45-60 minutos com ~36 interações, são vários minutos perdidos apenas manipulando o dispositivo.

Comunicação Celular ↔ Relógio

Comandos

Protocolo de comunicação

  • Celular → Relógio: Estado do treino (exercício atual, série, timer, progresso)
  • Relógio → Celular: Comandos (concluir série, pular exercício, voltar ao exercício pulado)
  • Tecnologia: Wear OS Message API / Data Layer API via react-native-watch-connectivity ou módulo nativo Kotlin
  • Fallback: Se a conexão cair, o relógio exibe "Reconectando..." e o celular continua funcionando normalmente

Métricas de Sucesso

Como sabemos que funcionou

Interações com celular

Antes: 36+ vezes tirando o celular do bolso por treino

Depois: Apenas para iniciar e finalizar o treino (2 interações)

Adesão ao timer

Antes: Timer ignorado por inconveniência de checar o celular

Depois: Countdown sempre visível no pulso + vibração ao fim

Tempo de treino

Meta: Redução de ~3-5 min por treino eliminando manipulação desnecessária do celular

Adoção

Meta: Watch companion se torna o método principal de acompanhamento — usado em >90% dos treinos

Riscos

O que pode dar errado

Estabilidade do Bluetooth

Alto

A conexão Bluetooth entre celular e relógio pode cair durante o treino, especialmente com interferência de outros dispositivos na academia.

Mitigação: Timer local no relógio que continua rodando independente da conexão. Reconexão automática com sync de estado. O celular continua funcionando normalmente como fallback.

Consumo de bateria do relógio

Médio

Manter a tela ativa e comunicação Bluetooth constante pode drenar a bateria do relógio rapidamente.

Mitigação: Usar ambient mode (always-on display simplificado) em vez de tela ativa constante. Enviar apenas deltas de estado, não o estado completo. Aproveitar o timer local para reduzir frequência de comunicação.

Complexidade do Wear OS

Médio

Desenvolver para Wear OS requer um app nativo (Kotlin + Jetpack Compose for Wear OS), que é um stack separado do React Native usado no Forja.

Mitigação: App do relógio é propositalmente simples — apenas display + 3 ações. Escopo mínimo reduz superfície de desenvolvimento. Módulo nativo no lado do celular para bridge com Wear OS Message API.

Tela circular e pequena

Baixo

A tela circular de ~1.4" do Galaxy Watch limita a quantidade de informação visível. Nomes longos de exercícios podem não caber.

Mitigação: Design focado em informação essencial — uma tela, uma ação. Truncar nomes longos com "..." e priorizar números (peso, série, timer) sobre texto.

Considerações Técnicas

Stack e implementação

App do relógio (Wear OS)

  • Linguagem: Kotlin
  • UI: Jetpack Compose for Wear OS (Compose Material 3 for Wear)
  • Comunicação: Wear OS Data Layer API (MessageClient para comandos, DataClient para estado)
  • Timer: CountDownTimer local — inicia com valor recebido do celular e roda independente
  • Haptics: Vibrator API do Android para feedback tátil
  • Ambient mode: Always-on display simplificado (apenas timer + nome do exercício)

Bridge no celular (React Native)

  • Módulo nativo: Kotlin module que expõe Wear OS MessageClient ao JS
  • Listener no workoutStore: Subscriber que envia estado ao relógio a cada mudança relevante
  • Comandos recebidos: Traduzidos para actions do workoutStore (completeSet, skipExercise, returnToSkipped)
  • Publicação: Wear OS app distribuído junto com o APK principal via Google Play (multi-module)

Formato dos dados trocados

Mensagens JSON mínimas para economizar banda Bluetooth:

  • Estado (celular → relógio): { exercise, setNum, totalSets, reps, weight, timerSec, totalExercises, currentIndex, status }
  • Comando (relógio → celular): { action: "completeSet" | "skip" | "returnToSkipped" }
Fora do Escopo (v1)

O que NÃO está incluído

  • Iniciar treino pelo relógio — seleção de plano e início continuam exclusivos do celular
  • Editar peso pelo relógio — input numérico em tela circular é impraticável; peso é registrado usando o último valor do celular
  • Histórico de treinos no relógio — consulta de histórico fica no celular
  • Standalone mode — o relógio não funciona sem o celular conectado; sem lógica de negócio local
  • Complicações / Tiles — sem widgets no watch face na v1; apenas o app completo
  • Apple Watch / outros wearables — v1 exclusiva para Samsung Galaxy Watch (Wear OS)
  • Notificações push no relógio — comunicação é apenas durante treino ativo via Data Layer
Fases

Plano de implementação

Fase 1 — Timer + Concluir série

P0

O mínimo viável para eliminar a maior fricção: ver o timer e concluir séries pelo relógio.

  • Bridge celular ↔ relógio (Message API)
  • Tela de exercício ativo (nome, série, peso, dots de progresso)
  • Tela de timer de descanso (countdown circular + haptic)
  • Botão "Concluir Série"
  • Auto-launch quando treino inicia
  • Tela idle quando sem treino ativo

Fase 2 — Navegação + Resiliência

P1

Funcionalidades complementares que completam a experiência.

  • Pular exercício (swipe ou botão)
  • Retomar exercícios pulados (checkpoint)
  • Reconexão automática com sync de estado
  • Ambient mode (always-on display simplificado)

Fase 3 — Polish

P2

Refinamentos de experiência para uso diário.

  • Animações e transições entre telas
  • Otimização de bateria
  • Tela de resumo ao fim do treino
